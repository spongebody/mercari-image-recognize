# 模型测试 (Model Testing) UI 与流程重设计 — 设计文档

- **日期**: 2026-06-09
- **范围**: 仅前端 (`web/` 目录)。不改动任何后端 API、数据存储或评测逻辑。
- **目标页面**: `/evaluations` (`web/evaluations.html`)

## 背景

"模型测试" 是 Mercari 图片识别系统的模型评测/基准框架。用户上传商品数据 (itemName / genreId / image / brand)，选择视觉模型、分类模型、商品数据模型与推理程度，后台批量跑模型推理，得到分类/品牌/标题预测与耗时，再由客服逐条人工校验 (正确/可接受/错误)，最后填写分析并归档。

当前实现是单一的 810 行 `web/evaluations.html`（内联样式 + 内联脚本），采用「左侧上传表单+运行记录、右侧状态/校验/分析」的单页堆叠布局。

## 问题 (用户确认)

四个方向均需改进，无单一痛点：
1. **视觉与质感** — 卡片/指标层级不够清晰。
2. **流程效率** — 无法复用历史配置重测；多轮之间无法对比。
3. **结果可读性** — 结果表太宽太挤、看不到商品图片、对错不直观、难定位出错条目。
4. **校验体验** — 客服逐条校验不够顺手（无批量、无筛选、无图片）。

使用者为 **测试者 + 客服分工**：测试者负责发起/监控/分析，客服专注逐条校验。

## 设计决策 (用户确认)

- **页面结构**: 分阶段 Tab（发起 → 监控 → 校验 → 分析），外加「多轮对比」入口。
- **图片缩略图**: 强需。校验区直接显示商品缩略图，点击放大。
- **迭代能力**: 需要「复用配置重测」与「多轮指标对比」。

## 关键约束: 仅前端

所有新功能复用现有 API，无需后端改动：
- **复用配置** — `GET /api/v1/evaluations` 已返回每个运行的 `visionModel / categoryModel / productDataModel / reasoningEffort / language` 等字段，直接回填表单。
- **缩略图** — 结果行的 `image` 字段已含图片 URL（可能多张，以 `|` 分隔）。
- **多轮对比** — 对所选每个运行分别调 `GET /api/v1/evaluations/{id}` 取 `summary`，前端并排渲染。

## 架构与代码组织

当前单文件加上新功能会膨胀到约 1500 行。按现有 `shell.js` / `shell.css` 共享资源的模式，拆分为：

- `web/evaluations.html` — 仅页面骨架与各 Tab 容器标记。
- `web/assets/evaluations.css` — 从内联 `<style>` 抽出的页面样式，扩展语义状态色。
- `web/assets/evaluations.js` — 状态与渲染逻辑，按 Tab 关注点组织（setup / monitor / review / analysis / compare）。

每个单元职责单一、通过明确的状态对象通信、可独立理解与（手动）验证。

## 信息架构 / Tab 职责

页面顶部为分阶段 Tab：`① 发起 · ② 监控 · ③ 校验 · ④ 分析`，右侧另有 `⇄ 多轮对比` 入口。左侧为常驻「运行记录」列表，带状态色块；选中某运行即载入其详情供各 Tab 渲染。

### ① 发起 (Setup)
- 上传测试数据 (CSV/TSV)，字段提示 itemName/genreId/image/brand。
- 模型配置：图片识别模型、分类模型、商品数据模型；推理程度 (none/low/medium/high)；语言 (ja/zh/en)；数量限制。
- **复用配置**：从任一历史运行一键回填模型/推理/语言（文件仍需重新选择）。
- 点击「开始测试」成功后自动切到 ② 监控。

### ② 监控 (Monitor)
- 实时进度条 (completed/total)、ETA、失败计数。
- 指标卡：分类准确率、品牌准确率、复核后分类/品牌、待校验数等（来自 `summary.overall`）。
- 运行中每 2500ms 轮询 `GET /api/v1/evaluations/{id}`；状态变 completed 后停止轮询并提示「去校验」。

### ③ 校验 (Review) — 客服主战场
- **筛选 chips**：全部 / 分类错 / 品牌错 / 待校验（纯前端基于已加载行过滤）。
- **缩略图列**：渲染 `image` 第一张为缩略图，点击打开 lightbox 放大（多图可在 lightbox 内切换）。
- **原→AI 对照**：分类、品牌列展示原值→AI 值，语义着色（正确=绿，待校验=琥珀，错误=红）。
- **逐条校验**：分类校验、品牌校验下拉 (待校验/正确/可接受/错误) + 备注。
- **批量标记**：选中多行后批量标为某一校验值。
- **键盘流**：↑/↓ 切换当前行，快捷键标记当前行分类/品牌校验值，加速逐条处理。
- 「保存校验」调 `PUT /api/v1/evaluations/{id}/review`，刷新统计。
- 归档后只读。

### ④ 分析 (Analysis)
- 可优化点 / 优化动作 / 下一轮测试建议 三个文本域，调 `PUT .../analysis`。
- 「带上这些建议复用配置重测」CTA：把当前运行配置回填到 ① 发起。
- 「下载结果 CSV」与「归档」(`POST .../archive`，归档锁定校验与分析)。

### ⇄ 多轮对比 (Compare)
- 勾选多个运行，并排展示指标表：分类/品牌准确率、复核后准确率、平均耗时等；同一指标的最优值高亮。
- 纯前端，对每个所选运行各取一次详情。

## 数据流与状态

单一 `evaluationState`（沿用现有结构）扩展：
- `activeTab` — 当前 Tab。
- `reviewFilter` — 校验区筛选条件。
- `selectedRows` — 校验批量操作选中集。
- `compareRunIds` — 对比所选运行集与其详情缓存。

选中运行后加载一次详情与结果，所有 Tab 从缓存状态渲染。轮询仅在监控阶段、针对 pending/running 状态启用。切 Tab 为纯客户端重渲染，不重新请求。

## 错误处理

- 沿用现有契约：API 错误用顶部内联 message 横幅展示。
- 每个 Tab 有空态/加载态文案。
- 运行未完成或已归档时，对应操作按钮禁用。
- Lightbox、复用配置、对比在数据缺失时优雅降级（不报错、给占位/提示）。
- 鉴权契约不变：未登录访问 `/evaluations` 仍重定向到 `/login?next=/evaluations`。

## 视觉方向

- 保留现有蓝色 accent 与共享 shell。
- 收紧卡片（更轻的阴影）、强化指标层级、统一校验状态语义色（绿/琥珀/红）。
- 校验表更紧凑、更易扫读。
- 无前端框架；vanilla CSS + 现有 CSS 变量，扩展状态色。

## 测试与验证

- **后端**：无改动，现有 Python API 测试 (`tests/test_evaluations_api.py`, `tests/test_evaluation_runs.py`, `tests/test_image_model_evaluation.py`) 保持通过。
- **前端**：用 Playwright 在浏览器对接真实/种子运行验证：页面加载、Tab 切换、校验筛选、复用配置回填、多轮对比、lightbox；并确认原有流程（创建/轮询/校验保存/分析/归档）不回归。
- （可选，待用户定）校验筛选/渲染等纯逻辑的 JS 单测——当前计划以浏览器验证为主。

## 非目标 (YAGNI)

- 不改后端 API、评测算法、数据存储格式。
- 不引入前端构建工具或框架。
- 不做多人实时协作/权限系统（角色仅靠 Tab 聚焦区分，不做硬隔离）。
